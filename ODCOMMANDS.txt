Now run the query to check blocking session.

SET LINESIZE 250
SET PAGESIZE 100
SET TRIMSPOOL ON
SET WRAP OFF

COL blocker_user   FORMAT A12
COL waiter_user    FORMAT A12
COL event          FORMAT A35
COL object_name    FORMAT A25
COL blocker_inst   FORMAT 999
COL waiter_inst    FORMAT 999
COL seconds_in_wait FORMAT 999999
SELECT
  s1.inst_id AS blocker_inst,
  s1.sid     AS blocker_sid,
  s1.serial# AS blocker_serial,
  s1.username AS blocker_user,
  s2.inst_id AS waiter_inst,
  s2.sid     AS waiter_sid,
  s2.serial# AS waiter_serial,
  s2.username AS waiter_user,
  s2.event,
  s2.seconds_in_wait,
  o.object_name
FROM gv$session s1
JOIN gv$session s2
  ON s1.sid = s2.blocking_session
  AND s1.inst_id = s2.blocking_instance
LEFT JOIN dba_objects o
  ON s2.row_wait_obj# = o.object_id
WHERE s1.username IS NOT NULL;

2) You discover that a datafile is missing from the file system. How will you recover it while the database is up and running?

=> If a datafile is missing while the database is open, that tablespace becomes offline automatically.

ðŸ”¹ Identify the missing file:
	SELECT file#, name FROM v$datafile WHERE status='RECOVER';

ðŸ”¹ Take the datafile offline:
	ALTER DATABASE DATAFILE '<path>' OFFLINE;

ðŸ”¹ Restore the missing datafile from backup:
	RMAN> RESTORE DATAFILE '<path>';

ðŸ”¹ Recover it:
	RMAN> RECOVER DATAFILE '<path>';

ðŸ”¹ Bring it online:
	ALTER DATABASE DATAFILE '<path>' ONLINE;


3) Your RMAN backup failed last night. How do you find the reason and ensure the next backup completes successfully?
=>
ðŸ”¹ Check RMAN logs:
	Review the RMAN output/log file from the scheduled job (via OEM, cron, or shell).
	Look for ORA- or RMAN- errors (e.g., ORA-19504, RMAN-03009).

ðŸ”¹ Check alert log:
	tail -50 alert_<SID>.log

ðŸ”¹ Query RMAN catalog/controlfile for failures:
	RMAN> LIST FAILURE;  
	RMAN> REPORT OBSOLETE; 

ðŸ”¹ Common causes:
	Archive destination full
	FRA full
	Channel allocation issues
	Missing backup destination permissions

ðŸ”¹ Fix and validate:
	Free up space / increase FRA size
	
ðŸ”¹ Test backup manually
	RMAN> BACKUP DATABASE PLUS ARCHIVELOG;
	Check log and confirm completion.
	RMAN> VALIDATE DATABASE;
	
	
4) A user accidentally dropped a critical table. How do you recover it without restoring the entire database?
=> If Flashback is enabled, you can recover it quickly without restoring the whole database.

ðŸ”¹Check if Flashback is enabled:
	SELECT FLASHBACK_ON FROM v$database;

ðŸ”¹Find the drop time:
	SELECT original_name, droptime FROM recyclebin WHERE original_name='TABLE_NAME';

ðŸ”¹Recover the table from Recycle Bin:
	FLASHBACK TABLE table_name TO BEFORE DROP;


5) Archive log destination is full and database hangs â€” what steps do you take?
=> When the archive log destination is full, the database can hang because it canâ€™t archive new redo logs.

ðŸ”¹Check archive destination:
	SHOW PARAMETER log_archive_dest;

ðŸ”¹Free up space or move old archives:
	Delete/move old archives to backup location:
	RMAN> DELETE ARCHIVELOG ALL COMPLETED BEFORE 'SYSDATE-2';

ðŸ”¹If urgent (DB hang):
	Temporarily change archive destination:
	ALTER SYSTEM SET log_archive_dest_1='LOCATION=/newpath' SCOPE=BOTH;

ðŸ”¹Resume database activity:
	Once space is available or destination changed, the archiver (ARCn) will continue writing, and DB will unhang automatically.


6) How will you perform a point-in-time recovery for a single tablespace using RMAN?
=> To recover one tablespace to an earlier point without affecting the rest of the DB, use RMAN TSPITR (Tablespace Point-In-Time Recovery).
ðŸ”¹Identify tablespace & SCN/time:

	SELECT tablespace_name FROM dba_tables WHERE table_name='TABLE_NAME';

ðŸ”¹Start RMAN and connect to target DB:

	rman target /
	
ðŸ”¹Run TSPITR command:

	RECOVER TABLESPACE users
	UNTIL TIME "TO_DATE('2025-11-10 10:00:00','YYYY-MM-DD HH24:MI:SS')"
	AUXILIARY DESTINATION '/u01/aux_dest';

ðŸ”¹RMAN creates an auxiliary instance automatically, restores datafiles for that tablespace from backup, applies redo till that time, and replaces the 
  recovered datafiles in the main DB.

ðŸ”¹Bring the tablespace online:

	ALTER TABLESPACE users ONLINE;


7) A query that was running fast yesterday is now taking hours. What steps do you follow to diagnose and fix it?
=> When a query suddenly becomes slow, I follow these steps:
ðŸ”¹ Get the SQL ID and check execution plan
ðŸ”¹ Check for missing or unusable indexes
ðŸ”¹ Check wait events (what the query is waiting on)
ðŸ”¹ Check bind variable peeking / histogram issues
ðŸ”¹ Fix based on root cause
ðŸ”¹ Monitor after fix

8) CPU usage is 100% on the DB server â€” how do you find which SQL or session is responsible?
=> When CPU reaches 100%, I first identify the top CPU-consuming sessions using v$session, v$process, and v$sesstat views. Then I check the associated SQL ID 
in v$sql to find which SQL statement is causing the spike. I verify if the issue is due to a bad execution plan, missing indexes, or sudden data growth. Based 
on the findings, I either tune the SQL, gather fresh statistics, or terminate the problematic session to stabilize the system.

9) Your primary database has gone down â€” how will you bring the standby database online as primary (Data Guard failover)?
=> When primary is down, I check the standby status, stop managed recovery, and run ALTER DATABASE FAILOVER TO <standby>.
   Then I open the standby as the new primary.
   If flashback is enabled, I can convert the old primary back to standby to restore Data Guard configuration.
   
10) Explain the steps to clone a production database to a test environment using RMAN.
=> To clone Prod to Test using RMAN, I take a full backup including archivelogs, transfer it to the Test server, restore the controlfile, mount the DB, catalog
   backups, then run RESTORE and RECOVER DATABASE. Finally, I open with RESETLOGS and optionally use NID to rename the DB.
